<Warning>
  The CompanyFlow API does not currently expose a direct DELETE endpoint for memos. Memos are typically archived rather than permanently deleted to maintain audit trails and compliance requirements.
</Warning>

## Archiving Memos

Instead of deleting memos, CompanyFlow uses an archive system where memos can be moved to an `archived` status. This preserves the memo data while removing it from active views.

### Status Transition

To archive a memo, you would typically update its status through your internal workflows or business logic:

```
Status: pending → archived
Status: approved → archived
Status: rejected → archived
```

## Alternative Approaches

<AccordionGroup>
  <Accordion title="Filter out archived memos">
    When listing memos, exclude archived memos by filtering:
    
    ```bash
    GET /memos?status=pending
    GET /memos?status=approved
    ```
    
    This returns only active memos without showing archived ones.
  </Accordion>

  <Accordion title="Query archived memos">
    To view archived memos specifically:
    
    ```bash
    GET /memos?status=archived
    ```
    
    This is useful for audit purposes or historical reporting.
  </Accordion>

  <Accordion title="Implement soft delete pattern">
    If you need to implement memo deletion in your application:
    
    1. Create a service layer method that updates memo status to `archived`
    2. Add a `deletedAt` timestamp field to track when archival occurred
    3. Implement business rules about who can archive memos
    4. Consider adding a retention policy for permanently removing old archived memos
  </Accordion>
</AccordionGroup>

## Data Retention

<Note>
  Memos contain important business communications and approval workflows. Consider implementing a data retention policy that:
  
  - Keeps active memos indefinitely
  - Archives memos after a certain period of inactivity
  - Permanently removes archived memos only after regulatory retention periods
  - Maintains approval history separately for compliance
</Note>

## Database Schema

The memos table includes a status field that supports archival:

```sql
status VARCHAR(50) DEFAULT 'pending' 
  CHECK (status IN ('draft', 'pending', 'approved', 'rejected', 'archived'))
```

This allows you to mark memos as archived without removing them from the database.

## Cascade Behavior

<Info>
  The database schema includes cascade delete rules:
  
  - If a company is deleted, all associated memos are deleted
  - If an employee is deleted, their created memos are deleted
  - Memo recipients and approval history are deleted when the memo is deleted
  
  These operations are handled at the database level via foreign key constraints.
</Info>

## Best Practices

## Implementation Example

If you need to implement memo archival in your application:

<CodeGroup>

```go Service Layer
func (s *MemoService) ArchiveMemo(ctx context.Context, memoID uuid.UUID, archivedBy uuid.UUID) error {
    // Check permissions
    if !s.canArchiveMemo(ctx, archivedBy) {
        return errors.New("unauthorized to archive memo")
    }
    
    // Update memo status
    _, err := s.db.ExecContext(ctx,
        "UPDATE memos SET status = 'archived', updated_at = NOW() WHERE id = $1",
        memoID,
    )
    
    // Log archival action
    s.logAudit(ctx, "memo_archived", memoID, archivedBy)
    
    return err
}
```

```sql Direct SQL
UPDATE memos 
SET status = 'archived', 
    updated_at = CURRENT_TIMESTAMP 
WHERE id = '123e4567-e89b-12d3-a456-426614174000';
```

</CodeGroup>

<Tip>
  Consider adding a `restore` endpoint that can transition archived memos back to their previous status if accidental archival occurs.
</Tip>
